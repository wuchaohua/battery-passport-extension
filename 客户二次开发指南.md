 # 电池护照平台 — 客户二次开发指南（二进制交付场景）

 ---

 ## 一、场景说明

 本指南面向 **天合储能** 和 **托邦股份** 的二次开发团队。
 平台方 **不交付 Java 后端源码**，交付的是混淆后的 jar 包 + Docker 镜像。

 客户方在二进制交付的前提下，通过以下途径实现业务定制：

 ```
 二次开发能力全景
 ┌──────────────────────────────────────────────────┐
 │                                                   │
 │  ① SPI 插件机制（Java）                           │
 │    → 校验器   变压器  报告定制  通知拦截  事件订阅   │
 │                                                   │
 │  ② RESTful API + SDK                             │
 │    → 开放数据接口  Java客户端SDK  OpenAPI文档       │
 │                                                   │
 │  ③ Webhook 事件回调                               │
 │    → 护照创建/更新  质量告警  报告就绪              │
 │                                                   │
 │  ④ 脚本引擎扩展（Groovy/JS）                      │
 │    → 校验规则  报告格式  数据增强                   │
 │                                                   │
 │  ⑤ 前端远程组件                                   │
 │    → 仪表盘扩展  品牌定制  新增页面                 │
 │                                                   │
 │  ⑥ 配置化定制                                     │
 │    → SSO适配  限流阈值  数据源  品牌色             │
 └──────────────────────────────────────────────────┘
 ```

 ## 二、六种二次开发方式详解

 ### 方式 ①：SPI 插件机制

 **适用场景**：在平台核心流程中插入自有业务逻辑

 **客户接触的内容**：`battery-plugin-api-1.0.0.jar`（含源码和javadoc的公开jar）

 **可实现的接口**：

 | SPI接口 | 触发时机 | 天合示例 | 托邦示例 |
 |---------|---------|---------|---------|
 | BatteryDataValidator | 电池数据导入/创建时 | 容量偏差校验 | - |
 | BatteryDataTransformer | 外部数据导入时(sourceType匹配) | - | ERP格式转换 |
 | ReportCustomizer | 报告生成后 | 天合Logo水印 | 托邦页脚模板 |
 | NotificationInterceptor | 通知发送前 | - | 企业微信推送 |
 | EventSubscriber | 各类状态变更事件 | 入库通知 | 质量告警 |

 **开发步骤**：

 ```bash
 # 1. 从平台Nexus获取依赖
 # (在 pom.xml 中声明)
 <dependency>
     <groupId>com.battery</groupId>
     <artifactId>battery-plugin-api</artifactId>
     <version>1.0.0</version>
     <scope>provided</scope>
 </dependency>

 # 2. 实现接口 (示例见 trina-custom-plugin/)
 # 3. 注册到 META-INF/services/ 文件
 # 4. 编译为 jar
 mvn clean package

 # 5. 提交给平台运维
 # 运维将jar放入 /opt/battery/plugins/{enterprise}/ 目录
 # 平台自动热加载，无需重启
 ```

 **关键设计——平台侧热加载机制**（对接的客户开发人员需理解）：

 ```java
 // 平台内部实现（不交付源码，但公开文档说明原理）
 public class PluginManager {

     private final Map<String, BatteryDataValidator> validators = new ConcurrentHashMap<>();

     @EventListener
     public void onPluginJarEvent(PluginJarEvent event) {
         // 1. 使用 URLClassLoader 从 plugins/ 目录加载 jar
         // 2. 通过 ServiceLoader 查找 SPI 实现
         // 3. 按 order() 排序注册
         // 4. 注入当前 enterprise code 进行过滤
     }

     public List<ValidateResult> validate(String enterpriseCode, BatteryData data) {
         return validators.values().stream()
             .filter(v -> matchesEnterprise(v, enterpriseCode))
             .sorted(Comparator.comparingInt(BatteryDataValidator::order))
             .map(v -> v.validate(data))
             .toList();
     }
 }
 ```

 ### 方式 ②：RESTful API + SDK

 **适用场景**：系统集成（ERP/MES对接、数据导出、第三方平台接入）

 **客户接触的内容**：OpenAPI 3.0 文档 + `battery-sdk-client-1.0.0.jar`

 **主要 API**：

 | 方法 | 端点 | 说明 |
 |------|------|------|
 | GET | /api/v1/passports/{id} | 获取电池护照详情 |
 | POST | /api/v1/passports | 创建电池护照 |
 | GET | /api/v1/passports | 列表查询（支持分页/筛选） |
 | GET | /api/v1/passports/{id}/trace | 获取溯源链 |
 | POST | /api/v1/passports/{id}/trace | 添加溯源事件 |
 | GET | /api/v1/reports/{id}/download | 下载报告PDF |
 | POST | /api/v1/webhooks | 注册Webhook回调 |

 **Java SDK 使用示例**：

 ```java
 BatteryPassportClient client = BatteryPassportClientBuilder
     .baseUrl("https://battery-api.trinasolar.com")
     .apiKey("key_abc123")
     .apiSecret("sec_xyz789")
     .build();

 // 查询护照
 PassportDTO passport = client.getPassport("BP-TR-2026-00001");

 // 生成报告
 byte[] pdf = client.generateReport("BP-TR-2026-00001", "COMPLIANCE", "zh-CN");

 // 注册 webhook
 client.registerWebhook(
     "https://customer.com/webhook/battery",
     "hmac-secret-123",
     List.of("PASSPORT_CREATED", "QUALITY_ALERT")
 );
 ```

 ### 方式 ③：Webhook 事件回调

 **适用场景**：客户自有系统异步接收平台事件

 **客户接触的内容**：平台配置界面的 Webhook 设置

 **事件类型**：

 | 事件类型 | 触发时机 | payload 包含 |
 |---------|---------|-------------|
 | PASSPORT_CREATED | 电池护照创建 | passportId, serialNumber, 完整数据 |
 | PASSPORT_UPDATED | 护照数据更新 | passportId, 变更字段 |
 | REPORT_GENERATED | 报告生成完成 | reportId, downloadUrl |
 | QUALITY_ALERT | 质量合规告警 | passportId, 告警级别, 详情 |
 | TRACE_UPDATED | 溯源事件新增 | passportId, 事件详情 |

 **实现示例**（Spring Boot）：

 ```java
 @PostMapping("/webhook/battery")
 public ResponseEntity<String> receive(@RequestBody Map<String, Object> payload,
                                        @RequestHeader("X-Webhook-Signature") String sig) {
     // 1. 验签
     if (!hmacVerify(payload, sharedSecret, sig)) {
         return ResponseEntity.status(401).body("invalid signature");
     }
     // 2. 处理事件
     dispatch(payload);
     return ResponseEntity.ok("received");
 }
 ```

 详细实现见 `customer-examples/webhook-receiver/`。

 if (enterpriseCode == "trina") {

 ## 五、插件开发规范

 ### 1. 接口兼容性

 - SPI 接口版本与平台大版本绑定（主版本号一致即可）
 - 接口方法变动会 100% 向后兼容（可加 default 方法）
 - 平台方会提前一个版本发布 API 弃用通知

 ### 2. 插件隔离

 - 每个插件使用独立 ClassLoader，类名冲突不会影响平台
 - 插件不能访问平台内部类（仅能访问 plugin-api 中的接口和 DTO）
 - 插件不能使用 platform 的内部依赖（如 MyBatis、Redis）

 ### 3. 性能约束

 - 每个校验器的执行时间不应超过 500ms
 - 插件禁止启动新线程或定时任务
 - 插件应尽量无状态（有状态需自行处理并发）

 ### 4. 异常处理

 - 校验器返回 ValidateResult 而非抛异常
 - 其他插件抛异常会被平台捕获，不影响主流程
 - 插件应记录自身日志（平台自动注入 Logger）

 ## 七、FAQ

 **Q: 我们想新增一个报表类型，不交付源码能实现吗？**

 A: 可以的。通过 ReportCustomizer SPI 或脚本引擎自定义报告格式。如果报表模板差异很大，也可以基于 REST API 拉取原始数据在客户自有系统中生成。

 **Q: 字段校验要改，每次都要平台发版吗？**

 A: 不需要。可以通过 Groovy 脚本引擎编写校验规则，热加载生效。或者实现 BatteryDataValidator SPI 插件，编译成 jar 后热部署。

 **Q: 我们想在前端加一个自己的页面，怎么办？**

 A: 平台支持远程组件加载。您将 React 组件打包发布到您的 CDN，在平台前端管理配置中注册挂载点和组件 URL 即可。平台前端框架会在对应位置动态加载并渲染您的组件。

 **Q: 插件能和平台一起做单元测试吗？**

 A: SDK 中提供了 Mock 工具类，您可以在本地编写 JUnit 测试，模拟 BatteryData 对象验证插件逻辑。平台侧的集成测试由平台方负责。

 **Q: 托邦的 ERP 数据格式变了怎么办？**

 A: 修改 TuobangErpDataTransformer.java 中的字段映射逻辑，重新编译 jar，提交给运维热部署即可，平台无需停机。

 **Q: 天合储能需要对接内部 CAS 系统怎么办？**

 A: 配置文件中设置 `battery.sso.provider = trina` 和 CAS 地址即可。如果 CAS 版本有特殊差异，可通知平台方扩展 TrinaSsoAdapter。

 **Q: 我们买了源码交付版本，上述方式还适用吗？**

 A: 适用。源码交付版本额外支持直接修改平台源码重新编译。但建议对核心逻辑仍走 SPI 插件机制，以便平台升级时您的定制逻辑无需大幅调整。

 **Q: 插件热部署失败怎么办？**

 A: 检查插件 jar 的 MANIFEST.MF 格式、SPI 文件路径是否正确、JDK 版本是否匹配。平台管理控制台会输出详细的插件加载日志。
